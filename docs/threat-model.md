# Damping ‚Äî Threat Model

> Phase 0.3 deliverable referenced by the original engineering plan but never written until now. This document doubles as the skeleton for the public-facing security whitepaper ‚Äî write it assuming an external security researcher or a skeptical HN commenter is the reader, not just internal engineering.

## 1. What Damping protects against

Damping sits between an AI coding agent (Claude Code, Cursor, and future integrations) and the real world ‚Äî the shell it executes commands in, and the MCP servers it calls tools on. The threat model is scoped to **actions with real-world side effects that the agent initiates**, not to the agent's reasoning quality or hallucination rate. Damping does not try to make the agent smarter; it tries to make sure a bad decision (the agent's own, or one it was manipulated into) doesn't execute unchecked.

### 1.1 What Damping is, and isn't: a safety net, not a security boundary

Stated explicitly here (2026-07-06, prompted by an external due-diligence review) because the rest of this document is honest about *individual* limitations (¬ß3's bypass-technique table, ¬ß6's fail-open reality, ¬ß8's tampering gaps) without ever stating the conclusion those limitations add up to: **Damping's detection is AST-informed pattern matching against a maintained rule set ‚Äî a denylist, structurally, even where the matching is done via a real parser instead of regex. This is a genuine, real-world-tested improvement for ¬ß2.1 below (a confused or careless agent, no adversary), but it is not, and cannot be, a reliable defense against ¬ß2.2/¬ß2.3 (a manipulated or adversarial agent actively trying to get past it).**

This isn't a Damping-specific weakness ‚Äî it's the documented, structural ceiling of this entire class of tool, confirmed by both academic and industry-red-team measurement in Damping's own product category: a 2026 study characterizing 1,709 real-world command denylists (13,332 rules total) found 69.0‚Äì98.6% of them bypassable by inputs their own authors intended to block (Chen & Lin, "One Goal, Many Commands: Characterizing Denylist Fragility in AI Agents," arXiv 2606.15549); a separate 2026 red-team study found shell-interception bypasses in 10 of 11 popular open-source AI coding/computer-use agents (Adversa AI, "GuardFall"). Anthropic's own stated strategy for containing Claude is to "design for containment at the environment layer first, then steer behavior at the model layer" ‚Äî i.e., sandboxes and OS-level isolation, not command-string policing, are the actual security boundary; command-level checks are a second, complementary layer at best. Simon Willison and Bruce Schneier have both made the same architectural argument publicly and independently.

**Practical implication for how to read the rest of this document:** every rule in `cli/policies/default.yaml` measurably helps against the failure mode that accounts for most real incidents (¬ß2.1) ‚Äî an agent that is not being actively attacked, just careless or overly broad in interpreting an instruction. Against ¬ß2.2/¬ß2.3, a sufficiently motivated adversary who knows Damping's rule set (all 28 rules are in a public repo) can very likely construct a bypass ‚Äî parameter-expansion-reconstructed command names, alternative interpreters/utilities not in the rule set, wrapper binaries, and similar techniques documented in the academic/red-team sources above are not hypothetical. Damping does not claim otherwise, and ¬ß4's self-protection guarantee is *tamper-evidence*, not tamper-prevention, for the same underlying reason (see ¬ß4's own wording). Treat Damping as one layer of defense-in-depth ‚Äî genuinely useful, honestly scoped ‚Äî not as a substitute for environment-level isolation (containers, sandboxes, least-privilege credentials) for anything security-critical.

## 2. Attacker / failure models

Damping must reason about three distinct classes, which are often conflated in casual discussion:

### 2.1 A confused or careless agent (no adversary)
The agent itself, acting in good faith on an ambiguous or overly broad instruction, issues a destructive command. This is the majority of the incidents above. Defense: the core interception + policy + confirm-prompt flow (Phase 1).

### 2.2 A manipulated agent (indirect prompt injection)
Content the agent reads (a file, a web page, an MCP tool's output) contains an instruction that causes the agent to take an unintended destructive action, or to poison its own memory so a *later*, seemingly unrelated action becomes dangerous (see ¬ß5, Memory & Context Poisoning). Defense: policy engine doesn't trust "the agent said it's fine" ‚Äî it evaluates the actual command/tool-call regardless of why the agent decided to issue it.

### 2.3 Damping itself as the target
Once an agent (or an attacker directing one) becomes aware a safety layer exists, the rational move is to disable it, not fight it. This is not hypothetical ‚Äî see the Ona incident referenced in the original planning docs, where an agent discovered a safety layer and simply turned it off. Damping's self-protection requirements (¬ß4) exist specifically for this class.

## 3. Known bypass techniques Damping must defend against (and test for)

Each of these must have a corresponding **"confirmed still blocked"** test case in `features/`, and any newly discovered bypass gets a new scenario added immediately ‚Äî permanent regression protection, never a one-off fix.

| Technique | Why naive detection misses it | Damping's defense |
| --- | --- | --- |
| Regex-evadable formatting (`r''m -rf`, extra whitespace, quoting tricks) | Regex matching on raw strings is trivially defeated by shell-legal formatting variance | AST-based parsing via mvdan/sh ‚Äî structure is normalized regardless of surface formatting |
| Multi-line script wrappers hiding `rm -rf` inside a larger heredoc/function | Naive single-line regex never sees the embedded line | mvdan/sh parses the full AST including function bodies, blocks, subshells, and if/while/for bodies; the danger-detection pass walks the whole tree, not just the top-level command. A heredoc body (`bash <<'EOF' ... EOF`) is additionally re-parsed and walked as its own script whenever it's addressed to a real shell interpreter (`sh`/`bash`/`zsh`/`dash`/`ksh`) ‚Äî a heredoc fed to a non-shell command (`cat`, `psql`, ...) is left alone, since that data is never executed as shell |
| Variable expansion / command substitution (`$(echo rm) -rf /`), including when it's used as an argument, redirect target, or here-string rather than the command name (`echo $(rm -rf ~)`, `cat <(rm -rf ~)`, `cat <<< "$(rm -rf ~)"`) | String matching doesn't resolve substitution structure; a substitution's embedded command executes at word-evaluation time regardless of how ‚Äî or whether ‚Äî its output is consumed | When a substitution supplies the command name itself, policy treats the dynamically-constructed command as at least `ask`-tier by default (can't prove intent statically). Independently of that, the AST walk descends into every `CmdSubst`/`ProcSubst` node wherever it appears (argument, assignment value, redirect target, here-string) and evaluates its embedded statements as their own script, so a resolvable destructive command inside one is caught directly by the normal rules (e.g. `destructive.rm_rf_protected`), not just swept into the generic unresolvable-command tier |
| Command wrappers that run another command as their arguments (`sudo rm -rf ~/`, `env FOO=1 rm -rf ~/`, `nohup`/`nice`/`setsid`/`timeout`/`exec`/`command`/`stdbuf`/`ionice`/`doas`/`time`) | Every rule dispatches on the command name, and the wrapper *is* the command name ‚Äî the real one sits inertly in its arguments | `cli/shell/facts.go`'s `commandPrefixes` table unwraps them (peeling stacked wrappers, skipping each wrapper's own flags, its value-taking flags, `env`'s `NAME=VALUE` assignments, and `timeout`'s leading duration operand) so `Facts.Command` is the command that actually runs. **This was a real, complete bypass of the entire rule set until 2026-07** ‚Äî `sudo rm -rf ~/` was allowed outright ‚Äî found by an adversarial review, now a permanent regression suite in `features/dangerous_command.feature`. Not unwrapped: `xargs`, because the operands arrive on stdin at runtime and no rule can judge `rm -rf` with no path at all ‚Äî a disclosed bypass needing a policy decision, not a parser change |
| A command hidden in an interpreter's `-c` string (`bash -c "rm -rf ~/"`, `bash -lc`, `sudo bash -c`) or handed to `eval` | The payload is one opaque string argument to a command named `sh`/`bash` | The `-c` script (including clustered spellings like `-lc`) and `eval`'s arguments are re-parsed and walked as their own script, exactly as heredoc bodies already were. Bounded by `maxReinterpretDepth` (8) so an adversarially nested payload cannot exhaust the stack ‚Äî past the cap Damping stops descending rather than crashing, a deliberate, disclosed tradeoff. Resolving these correctly also required fixing `cli/shell/literal.go` to apply the shell's own backslash-escape rules: mvdan/sh preserves a literal's *source* text, so `"sh -c \"rm -rf ~/\""` previously resolved to a string containing backslashes that the shell would never actually pass |
| Appending a harmless pipeline stage (`rm -rf ~/ \| cat`, `echo x \| sudo rm -rf ~/`) | A pipeline's `Facts` carries only the stage command *names* (`PipelineCmds`), never any stage's arguments, so every argument-inspecting rule saw nothing | Each pipeline stage is now walked as its own statement in addition to the whole-pipeline `Facts` the pipeline-shape rules (curl\|sh, base64\|sh, secret exfiltration) key off. Also a real, complete bypass until 2026-07 |
| Compound-command forms `time`, `coproc`, `case`, `declare`/`export`, `[[ ... ]]` | mvdan/sh gives each its own `Command` implementation rather than a `CallExpr`, and the AST walk's type switch handled none of them | `walkCmd` now descends into all of them (and `walkTestExpr` into `[[ ]]`'s expression tree), so `declare v=$(rm -rf ~/)` and `case x in a) rm -rf ~/ ;; esac` are caught. Found by the same review |
| Shell aliases resolving to dangerous commands | mvdan/sh's `syntax` package does **not** expand aliases (only the `interp` interpreter does, which means actually executing ‚Äî not acceptable for a pre-execution check) | An explicit alias-lookup table (`cli/shell/facts.go`'s `knownAliases`), resolved consistently whether the alias is a lone command or a pipeline stage. This is a small, deliberately extensible demonstration of the mechanism, **not** a comprehensive real-world alias database ‚Äî its one entry ("nuke") is a synthetic test fixture, not a claim that common dotfile-framework aliases are covered. Documented as a known limitation, not silently assumed solved by "we parse AST" |
| Base64/encoded payloads (`echo <b64> | base64 -d | sh`) | The dangerous content is runtime data, invisible to a static parser | Structural rule: any pipeline containing a decode primitive feeding into `sh`/`bash`/`zsh`/`eval`/`source` is flagged as at least `ask`-tier, regardless of decoded content. Covers `base64`/`base32`/`uudecode` by bare command name (unambiguous ‚Äî their only real-world purpose is decoding) plus `xxd -r`/`xxd -r -p` and `openssl enc -d`/`openssl base64 -d` by a targeted flag-pattern check against the raw text (their bare command names are ambiguous ‚Äî `xxd` alone just hex-dumps, `openssl` has dozens of unrelated subcommands, so matching on name alone would false-positive) |
| `/proc/self/root/...`, `/proc/self/exe` sandbox-bypass paths | Parser sees an opaque string literal, no special meaning | Maintained string-match list of known bypass paths, checked independently of AST semantics |
| `curl | sh` / `wget | sh` installer pipelines | Legitimate use exists (this is how Damping itself is installed!), so blanket blocking causes unacceptable false positives | Default policy treats `curl|sh` piping to `sh`/`bash`/`zsh` from a **non-allowlisted domain** as `ask`-tier; Damping's own install docs are the canonical example of what an allowlisted, user-initiated install pipeline looks like ‚Äî this asymmetry must be called out explicitly in the policy comments so contributors don't "fix" it into a hard block |
| Unicode homoglyphs / confusables in a command name (e.g. a fullwidth `ÔΩí` substituted for ASCII `r`) | Command-name matching (`Facts.Command` against rules and the alias table) is byte-exact string comparison with no Unicode normalization step | **Not currently defended against ‚Äî documented as a known limitation, not silently assumed solved.** A real POSIX shell wouldn't resolve most confusable byte sequences to the real binary either, so this is lower-severity than the other rows here, but it's a known LLM-prompt-injection evasion technique worth tracking if it's ever seen used against this class of tool |

## 4. Damping cannot be silently disabled (self-protection)

Requirement, not aspiration: an agent operating inside a session must not be able to turn off enforcement without the *human* being made aware. **This is a tamper-evidence guarantee, not a tamper-prevention one** ‚Äî per ¬ß1.1, a host already running arbitrary code from an adversarial or compromised agent cannot be made to reliably self-report every possible way it was interfered with; the honest claim is "silent disablement gets noticed and logged," not "silent disablement is impossible." Concretely:

- Policy file and hook registration in `~/.claude/settings.json` / `.cursor/hooks.json` / `~/.codex/hooks.json` should be checked for tampering by `damping doctor`, which is designed to be run frequently and as part of `damping init`'s final step.
- `damping off` is the only sanctioned way to disable enforcement, and it is a deliberate, explicit CLI invocation by a human at a terminal ‚Äî not something reachable via a Bash tool call the agent itself can issue, because the CLI binary itself is not typically on the list of commands an agent would be instructed to run for a legitimate reason, and `damping off` prints a loud, hard-to-miss warning plus writes an audit entry with `action_type: self_disable`.
- If Damping detects its own hook entry has been removed from the agent's settings file since the last `damping doctor` run (i.e., something *other* than `damping off` disabled it), it surfaces this loudly on the next `damping doctor` invocation ‚Äî this is the direct, named mitigation for the Ona-style "agent discovers and removes its own safety layer" failure mode. This stateful before/after comparison is `damping doctor`'s job specifically (it persists `doctorState` between runs ‚Äî see `cli/cmd/doctor.go`); `damping status` is a quick, stateless, current-instant summary and does not track or flag this on its own ‚Äî run `damping doctor` for the actual security check.

## 5. Adjacent threat class tracked for later phases: Memory & Context Poisoning (ASI06)

Not in scope for Phase 1-3, but documented here because Tim has flagged it as the most actively interesting extension (see `Â∏ÇÂÝ¥Ë™øÊü•ËàáÁèæÊ≥ÅÁ∏ΩË¶Ω.md` ¬ß11) and the same `core/policy` + `core/audit` spine is meant to extend into it in Phase 6 without a redesign. Real, verified threat data (not speculative):
- MINJA: memory poisoning via normal conversational interaction, 95-98% success rate.
- AgentPoison: <0.1% poisoning rate achieves >80% attack success, no retraining needed.
- PoisonedRAG: 5 malicious documents against a million-document corpus achieves 91-99% attack success.
- GoPlus Security's AgentGuard team disclosed this exact attack *pattern* as an illustrative, hypothetical example when launching their product (an agent induced to "remember a preference" ‚Äî e.g. "usually proactively refund" ‚Äî later triggered into unauthorized fund transfers via a vague "handle it as usual" instruction; see ChainCatcher, 2026-05-15, and Bitget News' independent coverage of the same disclosure). **Correction (2026-07-06)**: this was previously mischaracterized here as a "real financial-sector case" ‚Äî both primary sources frame it explicitly as a hypothetical/forward-looking illustration tied to a product launch, not a confirmed real-world incident with a named victim. Caught by an external due-diligence review and independently re-verified before this line was fixed ‚Äî the underlying attack technique is still worth taking seriously (it matches MINJA/AgentPoison's general mechanism), the citation just shouldn't claim more certainty than the source supports.

When Phase 6 starts, `ActionEvent.ActionType = memory_write` (already reserved in the schema, see `docs/architecture.md` ¬ß3) becomes a first-class interception point, with a provenance/trust score attached rather than the binary allow/deny used for shell/tool actions.

## 6. Fail-open vs. fail-closed ‚Äî an explicit, non-negotiable design split

Two different questions get conflated if this isn't stated explicitly:

1. **"Is this action dangerous?"** ‚Äî the actual policy judgment. Default posture: conservative, prefer under-blocking to over-annoying (per the "false positives are the #1 enemy" principle), because a tool that nags constantly gets uninstalled, and an uninstalled tool protects nobody.
2. **"Damping itself just failed internally (parser crash, corrupt policy YAML, hook process timeout) ‚Äî what happens to the action?"** ‚Äî This is answered by forces outside Damping's control: both Claude Code and Cursor's hook contracts fail open on anything other than exit code 2 (see `docs/cli-reference.md` ¬ßHook Contract). Damping cannot force a fail-closed outcome here even if it wanted to ‚Äî the surrounding agent decides. Damping's actual responsibility in this failure mode is to make the degradation **loud and logged**, not to pretend it can silently guarantee fail-closed: every internal failure writes an audit record with `decision.degraded = true`, and `damping doctor` treats any degraded event in the last 7 days as a hard warning, not a footnote ‚Äî a rolling window, not a "since your last `doctor` run" comparison (unlike the hook-registration and policy-hash checks in the same command, which do track state between runs via `doctorState`; a degraded event from 3 days ago will keep showing up on every `doctor` invocation until it ages out of the window, not just the first one after it happened).

**A related case that fits the same principle even though it's not an internal failure: an unrecognized agent's hook payload.** `cli/cmd/hook.go`'s `runHook` recognizes three agents today, by switching on the payload's own `hook_event_name` field ‚Äî Claude Code's and Codex's shared `PreToolUse` value (the two are then told apart by `turn_id`/`tool_use_id`, present only in Codex's payload ‚Äî see `docs/cli-reference.md` ¬ß11) and Cursor's `beforeShellExecution`. Any other value is routed through the same `logDegraded` path as a malformed payload ‚Äî no policy evaluation happens (Damping can't judge a contract shape it doesn't parse), and the action fails open the same way it would for any recognized agent's own internal failure, but it is never a silent no-op: `damping doctor` will surface it exactly like any other degraded event. This matters concretely for the MCP channel by contrast: `cli/adapter/mcp` has no agent-identifying branching at all (any MCP-speaking client is evaluated identically), so it isn't exposed to this particular failure mode in the first place.

## 7. Confused deputy (Phase 3, MCP Gateway)

Documented here for continuity even though it's a Phase 3 concern: the Gateway must never forward a client's MCP token directly to a downstream server. Each downstream MCP server gets its own audience-bound token, re-issued by the Gateway after the client's original token is validated ‚Äî this prevents a token scoped to Server A from being replayed against Server B. V1's thin MCP adapter (`cli/adapter/mcp`) does not handle tokens at all (see `docs/architecture.md` ¬ß7), so this class does not apply until Phase 3.

## 8. Policy tampering

The policy YAML file itself is a target ‚Äî if an agent (or an attacker-controlled process) can rewrite `~/.damping/policy.yaml` to remove a rule, the rule might as well not exist. V1 mitigation: `damping doctor` computes and displays a hash of the active policy file and flags unexpected changes since the last run (informational in V1 ‚Äî the user is the root of trust on their own machine, so this is a tamper-evidence measure, not tamper-prevention). Phase 5 (enterprise, centrally-managed policy) upgrades this to signed policy bundles distributed from the team/enterprise control plane, where tamper-evidence becomes tamper-*resistance* because the local user is no longer the sole trust root.

Writes to the policy file itself (`core/policy.AppendAlwaysPattern`, used by the `[A]`/`[D]` prompt responses) go through a temp-file-then-rename, so a crash mid-write can never leave a truncated or corrupt file behind ‚Äî but this does not serialize two *concurrent* writers (e.g. the CLI hook and `damping mcp wrap` both persisting a pattern at the same instant); the later rename simply wins. Given how rarely two such writes land in the same instant on one developer's machine, this is treated as an acceptable known gap rather than something requiring a cross-process file lock in V1.

## 9. Concurrent TTY prompts within one process

A single long-running `damping mcp wrap` process can have the MCP SDK dispatch multiple simultaneous tool calls, each potentially needing a `Prompt`-tier confirmation. `cli/adapter/mcp`'s `ttyPromptMu` mutex serializes these so two prompts never interleave their text/input on the same terminal within that one process. This does **not** cover a genuinely separate `damping` process (e.g. the CLI hook firing at the same instant as `mcp wrap`) prompting on the same terminal ‚Äî that cross-process race remains a known, low-priority limitation, consistent with ¬ß8's treatment of concurrent policy-file writes.

## 10. `damping dashboard`'s local HTTP surface

`damping dashboard` (`cli/dashboard`, see `docs/cli-reference.md` ¬ß9.1) is a deliberate exception to "every action goes through the same policy engine" ‚Äî it's not an interception point, but it's still the one place this project runs an actual network listener, so its own threat model is written out explicitly rather than assumed. **As of the 2026-07 self-update wiring, it is no longer purely a read-only viewer** ‚Äî `POST /api/update` can now modify the installed `damping` binary itself, a materially different capability from every other route here, which only ever reads `~/.damping/audit.jsonl`/`policy.yaml`; see the dedicated callout below rather than treating it as one more bullet alongside the read-only routes.

- **No authentication, by design.** This is a local single-user tool, not Phase 4's team dashboard (¬ßPhase 4 elsewhere in this doc's companion planning docs) ‚Äî adding auth would imply a multi-user trust boundary that doesn't exist here. The accepted threat model is the same as any other unauthenticated local dev server (`pprof`, most webpack dev servers, Prometheus's default config): anything that can already run code as the same OS user can already read `~/.damping/audit.jsonl` directly, so a local, unauthenticated HTTP view of the same file adds no new capability to that attacker.
- **DNS rebinding is explicitly defended against.** Binding to `127.0.0.1` alone does not stop a malicious webpage from reading this server: the browser can resolve an attacker-controlled domain to `127.0.0.1` mid-session and then treat the connection as same-origin with that domain, defeating the browser's own same-origin policy protection a *different*, unrelated origin would otherwise get. `cli/dashboard/server.go`'s Host-header check closes this ‚Äî a rebound request still carries the attacker's domain as its `Host` header, which the check rejects. This check only applies while bound to the default `127.0.0.1`/`localhost`; an operator who explicitly passes `--host` something else has already opted into a broader, explicitly-warned trust boundary (see `cli/cmd/dashboard.go`), and there is no longer one correct Host value to allowlist.
- **Not defended against, and out of scope**: another local OS user on a shared multi-user machine connecting to the same loopback port ‚Äî POSIX loopback sockets aren't user-isolated by default, so a second unprivileged account on the same box could `curl 127.0.0.1:4243` too. This is an accepted limitation shared by the same class of local dev tools referenced above, not something this project tries to solve; a genuinely multi-tenant machine has bigger problems than one more readable local audit log.

**The one outbound network call Damping's own CLI/dashboard makes on its own initiative.** Every other network activity in this project is either something the user explicitly asked for (installing, `damping mcp wrap` talking to the real MCP server it wraps) or entirely local (the dashboard's own HTTP listener above). The one exception: `cli/update.Check` ‚Äî invoked once, at the end of `damping init`/`status`/`doctor`/`dashboard`'s own output, and again by `GET /api/version` on the dashboard's behalf (`POST /api/update` itself performs no fresh check ‚Äî it just re-runs the self-update command for whichever update `GET /api/version` already reported) ‚Äî issues a single `GET` to GitHub's public releases API (`api.github.com/repos/amplify-lab/damping/releases/latest`) to learn the latest published version tag. No user data, filesystem content, audit history, or policy content is sent ‚Äî the request carries only a fixed `User-Agent` header GitHub's API requires. The response (just a version tag) is cached locally (`~/.damping/update-check.json`, or `$DAMPING_HOME/update-check.json`) and reused for up to 24h, bounded by a 1.5s timeout, with every failure mode swallowed rather than surfaced. Set `DAMPING_NO_UPDATE_CHECK` to any non-empty value to disable this call entirely ‚Äî no cache read, no network, on any command. See `docs/cli-reference.md` ¬ß6.1 for the full behavior this backs.

**`POST /api/update`'s CSRF exposure and defense ‚Äî a genuinely new class of risk for this dashboard.** Every route documented above only ever reads local files; this one can overwrite the installed `damping` binary in place. Neither of this section's existing defenses closes the gap on their own: the dashboard has no authentication at all by design, and the Host-header check defends specifically against DNS rebinding, not against a plain cross-origin request with a correctly-matching Host header. Concretely, without an additional defense, a malicious webpage open in the same browser at the same time `damping dashboard` happens to be running could fire a blind cross-origin `fetch`/form `POST` straight at `http://127.0.0.1:4243/api/update` ‚Äî no DNS rebinding needed, since a same-origin `Host: 127.0.0.1` header requires no trickery to produce ‚Äî and a simple cross-origin `POST` triggers no CORS preflight on its own, so the browser would send it. **Defense, part 1 ‚Äî the header, against a malicious browser tab**: `POST /api/update` requires a custom `X-Damping-Dashboard: 1` request header, checked immediately after the loopback check below and still well before method detection, the in-flight guard, or any streaming response starts; a request without it gets `403 Forbidden` immediately. A custom header forces the browser to CORS-preflight the request first, and because this server never sends an `Access-Control-Allow-Origin` header, the browser's own same-origin policy fails that preflight and the real `POST` is never sent for any cross-origin caller ‚Äî the same pattern Vite's and webpack-dev-server's local dev servers rely on to protect themselves.

**Defense, part 2 ‚Äî loopback-only, against a plain `curl` from elsewhere on the network**: the header alone stops a *webpage*, but does nothing against a bare `curl` from another machine, which can set any header it likes with no browser or CORS involved at all ‚Äî the exact gap an operator opens by running `damping dashboard --host 0.0.0.0` (¬ß10's DNS-rebinding bullet above already covers why that's a real, explicitly-warned-about choice). So, independent of and in addition to the header check, `POST /api/update` also requires the request's actual TCP peer address (`http.Request.RemoteAddr`, never a spoofable header) to be loopback. Unlike the DNS-rebinding Host-header check above ‚Äî which deliberately steps aside once `--host` is bound to something other than `127.0.0.1`/`localhost`, since there's no longer one correct value to allowlist ‚Äî this loopback check runs unconditionally regardless of `--host`: a stranger elsewhere on the LAN reaching an endpoint that overwrites the installed binary is unacceptable independent of what the operator bound the read-only routes to.

Together, the two checks run first (loopback check, then header check) and are what actually stop this endpoint from being reachable by anything other than a legitimate same-machine dashboard client: the endpoint's other safety checks (re-validating that the install location doesn't need elevated privileges, an in-flight guard against a second concurrent update) are real correctness backstops, but they run *after* both of these and do not by themselves stop an arbitrary web page or remote `curl` from reaching the endpoint at all. See `cli/dashboard/handlers.go`'s `remoteAddrIsLoopback`/`dashboardHeader`/`handleUpdate` doc comments for the implementation and `docs/cli-reference.md` ¬ß9.1 for the endpoint's full contract.
