## 一、本文件解決的核心衝突

接手後逐字讀完五份文件，發現一個**真正對不上、而且是路線層級的衝突**，其餘多是命名沿革（已自然收斂，見下方§二），不需要裁定。

### 衝突內容

| 文件 | 立場 |
| --- | --- |
| `總體開發藍圖-Fable5接手版.md`（章一、章五） | V1（衝 Product Hunt 的第一個公開版）**只做 CLI**，明文寫「不做：...MCP...全部 V2 以後」。理由：MCP Gateway 已是紅海（13+ 對手），CLI 安全層是唯一沒被覆蓋的縫，要用「週」為單位交付。此文件並自我宣稱「當本文件與先前規劃衝突時，以本文件為準」 |
| `開發計畫.md`（【修訂 A】） | 推翻上面這條，明文寫「排序修訂結論：MCP 通道從原 Phase 3 提前，與 CLI 一起併入第一個公開版...**覆蓋前文 Phase 1/Phase 3 的舊排序**」。理由：純 CLI 攔截「重複造輪子且傷害定位」，第一個公開版至少要做到「CLI + MCP 同引擎同稽核」的最小差異化門檻 |

兩份文件都用「以我為準」的語氣覆蓋對方，但沒有互相引用、互相意識到對方的存在——這是典型的「兩次獨立會議做了兩個互斥決定，沒人把兩份會議記錄放在一起看」。這正是使用者要我抓出來的「對不上的地方」。

### 裁定所需的事實查核（已完成，見研究摘要）

裁定前，我針對「純 CLI 攔截在 2026 年是否還是空白利基」做了實際競品調查，而不是憑印象各打五十大板。關鍵發現：

1. **純 CLI 攔截這個利基，事實上已經不是空白——已有成熟對手**：`Destructive Command Guard (dcg)`（Dicklesworthstone/destructive_command_guard，Rust 寫的），**1,150+ GitHub stars、1,751 commits、每日仍在更新、支援 10+ 種 Agent（Claude Code、Codex、Gemini CLI、Copilot CLI、Cursor、Grok、Aider）、50+ 安全規則包**。它用 RegexSet 篩選 + heredoc/腳本抽取 + tree-sitter/ast-grep AST 比對的三層架構偵測 `rm -rf`、`git reset --hard`、`DROP TABLE` 等，TOML 設定檔，本身就是這個賽道目前最成熟的專案。如果 Damping V1 只做「CLI 攔截危險指令」，在 Product Hunt/HN 上會被直接拿去跟 dcg 比較，而 dcg 的規模與涵蓋廣度會讓「又一個攔截器」成為公允的第一反應。
2. **原生威脅正在兌現，不只是總體開發藍圖風險清單裡的假設**：Anthropic 已於 2026 年 3 月在 Claude Code 內建「auto mode」——一個內建分類器，自動核准/拒絕風險指令，不需要第三方 hook 也能做到部分同樣的事。Cursor 也已出貨官方 Hooks API（`beforeShellExecution`、`beforeMCPExecution` 等）。這代表「攔截點本身」正在被各 Agent 廠商原生商品化——第三方工具如果只靠「我也接在同一個攔截點上做規則比對」，護城河只會越來越薄，不會越來越厚。
3. **反過來，「CLI + MCP 同一引擎同一稽核」這件事，目前查到的對手裡沒有一個真正做到**：dcg / sh-guard / claude-guardrails / agent-guardrails 都只做 CLI 端；Pipelock（740 stars）、Aegis（362 stars）偏向 MCP/HTTP 流量的閘道與防護，但不是「同一個政策引擎、同一份稽核紀錄橫跨 CLI 與 MCP」的設計。這正是修訂 A 所主張的差異化，而且現在有數據支持，不是空想。

### 裁定結論（本文件定案，覆蓋前兩份文件的排序）

**V1（衝 Product Hunt 的第一個公開版）= CLI 攔截 + 一個「輕量、非企業級」的 MCP 政策/稽核轉接器，兩者共用同一個 `core/policy` 引擎與 `core/audit` 出口。但 V1 的 MCP 轉接器範圍嚴格限縮，不包含 OAuth 2.1、confused deputy 防禦、audience 綁定這些企業級機制——那些留在原本規劃的 Phase 3。**

這不是各打五十大板的和稀泥，而是把「MCP 轉接器」拆成兩個明確不同的交付物，分別放進兩個不同 phase：

| | V1 的 MCP 轉接器（本文件新定義） | Phase 3 的 MCP 轉接器（原規劃不變） |
| --- | --- | --- |
| 做什麼 | 攔截 MCP client 端發出的 tool call，丟進 `core/policy` 判斷、寫進 `core/audit`。**沒有身分驗證、沒有 token 轉發、不是獨立 gateway process**，就是「幫 MCP 呼叫也套一層跟 CLI 一樣的政策+稽核」 | 獨立部署的 Gateway（可作為 reverse proxy 擋在真正的 MCP Server 前面）、OAuth 2.1 + PKCE、confused deputy 防禦（不轉發 client token，換發窄範圍新 token）、audience 綁定、多租戶 |
| 技術量級 | 官方 MCP Go SDK 目前**沒有內建攔截/中介層 hook**（見下方技術修正），要接在 client 端 tool-call 分派點上手動包一層——實測是「幾天」等級的工程，不是「幾週」 | 官方 SDK 的 `auth`/`oauthex` 套件雖已到 v1.5.0 穩定，但要蓋一個真正的 reverse-proxy gateway、做 audience 驗證與 token 重新簽發，這是「幾週」等級的工程，維持在 Phase 3 |
| 對外故事 | 「同一顆大腦、同一份稽核，橫跨你的終端機和你的 MCP Server」——這是 V1 就能在 demo 裡真實展示、跟 dcg 拉開差距的東西 | 「銀行/企業級的身分治理」——留給 Phase 5 銀行客戶用 |

**時程調整**：原 Phase 1 估「4-6 週」，加上這個輕量 MCP 轉接器後調整為 **5-8 週**，仍是「週」為單位，沒有違背總體開發藍圖「用週交付」的鐵律。

**對外文案調整（重要，寫進 README 與 launch 文案）**：不要領著「我們攔截 rm -rf」出場——這個標題 dcg 已經佔住,若正面比較新產品規模與历史都居下風。demo GIF 仍然可以用「rm -rf 被攔下」開場（視覺震撼、真實事件感強，適合當 hook），但**收尾的賣點必須是「一個政策、一份稽核，通吃 CLI 與 MCP」**，並且在 README 直接寫一段「與 dcg 等工具的差異」比較表（誠實列出對方做得好的地方,再指出「但它不管你的 MCP 呼叫」）——這是成熟 OSS 安全工具的標準做法（誠實比較比含糊帶過更容易在 HN/PH 上站得住）。

## 二、命名：本文件確認的最終狀態（非新裁定,是收斂記錄）

命名沿革本身**沒有衝突**，只是分散在多份文件裡容易誤讀成「還沒定案」。這裡收斂成一句話：

**產品只有一個名字：Damping（阻尼）。「Leash」「Corral」「Shepherd」「Rein」「MCPWarden」全部是命名沿革中的歷史代號，全部作廢，全域替換為 Damping。** 產品線 A（個人/開源/免費）與產品線 B（企業/主權）不是兩個名字，是同一個 Damping 的兩個入口：「Damping」（免費版）與「Damping Enterprise」（企業版）。CLI 執行檔叫 `damping`。

### 命名查核補充（本文件新增,總體開發藍圖與品牌文件當時尚未查證的部分）

品牌文件列出「待 Tim 完成的查核」，本文件用網路搜尋做了best-effort初步查核（**非正式商標意見，Tim仍須完成 USPTO TESS、GitHub org 即時查詢、網域註冊商 WHOIS 三項正式確認**）：

- **npm / PyPI**：`damping` 裸名皆為 404,未被佔用，與品牌文件記錄一致。
- **GitHub**：`github.com/damping` 這個帳號/組織 handle **確實存在**（ID 25096907），但底下零公開 repo、零 star、看起來是閒置/被搶注但未使用的帳號。**這件事有實際後果**：GitHub 的 org 名稱與使用者名稱共用同一個命名空間,一個已存在（即使閒置）的帳號通常無法被直接註冊為新 org,除非走 GitHub 的使用者名稱爭議流程（且不保證成功）。**建議 Tim 不要賭「拿到 `github.com/damping` 這個 org」**,改用母品牌當 org 名稱。
- **域名**：`damping.dev`、`getdamping.com` 查無 DNS 解析,看起來可註冊。`damping.com` 已被一家做「噪音與振動控制工程」的公司（Damping Technologies, Inc.）註冊使用中——產業完全不同、不構成商標衝突疑慮,但代表拿不到最順的裸 `.com`,這對一個開發者工具其實無妨（`.dev` 本來就更貼合受眾）。
- **建議**：GitHub org 用 **`amplify-lab`**（對應母品牌「增幅實驗室」），repo 叫 `damping`，即 `github.com/amplify-lab/damping`。這樣做還有一個額外好處：品牌文件明言「增幅實驗室後續產品皆以物理術語命名」,未來的第二、第三個產品可以放進同一個 org 底下,不用每次都賭一個新的裸名 org 是否可搶到。域名主推 `damping.dev`。

### 技術修正一：mvdan/sh 不是萬能的,AST解析有明確邊界

原`開發計畫.md`寫「一定要parse AST才能可靠判斷...不可以用regex猜」,這個方向正確但**過度簡化**。研究確認 mvdan/sh 的 `syntax` 套件是純**語法**解析器,以下三類繞過手法**AST本身解決不了,必須額外寫語意分析層**,已寫進`docs/threat-model.md`與`features/`：

1. **別名展開（alias）**：`syntax`套件不處理alias,只有`interp`（直譯執行）套件才會展開——但執行直譯本身有安全疑慮(等於真的跑一遍)。做法：維護「已知危險指令的常見別名」比對清單,而非依賴自動展開。
2. **編碼payload（如 `echo <base64> | base64 -d | sh`）**：AST能正確解析出這是一個pipeline,但看不懂被base64包住的內容是什麼(那是執行期才展開的資料,不是語法)。做法：偵測「pipeline中出現 `base64 -d`／`base64 --decode` 後接 `sh`/`bash`/`zsh`/`eval`」這個**結構性pattern**本身就該觸發至少`prompt`層級的攔截,不需要真的解出base64內容。
3. **`/proc/self/...`路徑**：對parser而言只是一段字串,沒有特殊語意。做法：維護已知繞過路徑的字串比對清單(如`/proc/self/root/`、`/proc/self/exe`),當作獨立規則,不指望AST自動理解。

**結論寫法**：AST解析解決的是「防止用regex猜測時被『多一個空格』『用變數展開』『多行包裝』這類語法花招唬弄」,不是解決「所有繞過手法」——語意層的已知繞過pattern仍要單獨列規則、單獨測試。這個認知差異已經反映進threat-model.md與features/。

### 技術修正二：Claude Code Hook 的真實合約（原文件沒寫清楚,直接影響 `cli/adapter/hook` 的實作）

- Hook透過`~/.claude/settings.json`（全域）或`.claude/settings.json`/`.claude/settings.local.json`（專案）的`hooks.PreToolUse[]`註冊,每筆設定含`matcher`（比對tool name,如`"Bash"`）與`hooks`陣列。
- Damping的hook腳本從stdin收到JSON,關鍵欄位：`session_id`、`cwd`、`hook_event_name`（值為`"PreToolUse"`）、`tool_name`、`tool_input.command`（實際指令字串）。
- **決策回傳方式,原文件沒寫對**：不是「任何非零結束碼都算攔截」。真正的合約是——
  - **Exit code `2`** = 攔截(blocking),Claude Code取消該次工具呼叫,stderr內容會回饋給模型當作錯誤訊息。
  - **Exit code `0`** + stdout輸出JSON,其中`hookSpecificOutput.permissionDecision`可為`"allow"`/`"deny"`/`"ask"`/`"defer"`,並可帶`permissionDecisionReason`——**這是Damping要跳出TTY確認提示(ask)時該用的路徑**,單純exit code無法表達「問使用者」這個狀態。
  - 其他非零結束碼(1、3...)是**non-blocking**——工具呼叫照樣執行,只有stderr第一行會顯示在transcript,**不能拿來當攔截機制**,這是原規劃裡容易寫錯的地方。
- Cursor的對應機制是`beforeShellExecution`(連同`beforeMCPExecution`、`beforeReadFile`、`afterFileEdit`等),設定在`.cursor/hooks.json`,stdin收到`{command, cwd, sandbox, ...}`,回傳`{permission: "allow"|"deny"|"ask", user_message, agent_message}`。同樣是exit code `2`才算攔截,其餘**預設fail-open**(除非設定`failClosed: true`)。**這代表Damping的hook腳本本身若crash或逾時,兩家Agent都預設放行(fail-open)**——這件事必須在threat-model與doctor設計裡正面處理(見下方§六「失敗模式」)。

### 技術修正三：官方 MCP Go SDK 沒有內建攔截點

原規劃隱含假設「SDK應該有中介層/interceptor可以掛」。查證結果：**沒有**。SDK的設計是`mcp.AddTool(server, ...)`這種「註冊工具」模式,不是「攔截既有呼叫」模式。這代表：

- **V1的輕量MCP轉接器**：做法是在**client端**包一層——Damping提供一個`damping mcp wrap <server-command>`包裝器,或提供一個實作MCP client介面的library hook,在tool call真正送出前先過`core/policy`。這是V1範圍。
- **Phase 3的正式Gateway**：做法是**自己寫一個完整的MCP server**去對接原本的MCP client,自己再轉呼叫真正的下游MCP Server(即reverse proxy模式)——這才是原規劃「gateway」的真正工程量,SDK不會幫你省這一步。

## 六、跨Phase的補充原則(本文件新增,原文件未涵蓋)

### 失敗模式設計原則(Fail-Open vs Fail-Closed 要分開講,不能混為一談)

- **政策判斷本身的結果**(這條指令危不危險)：寧可保守攔多一點、寧可多問一次——這是原文件已定的「誤判是頭號敵人但寧可少攔不可亂攔」,維持不變,這條講的是「規則要不要觸發」。
- **Damping自己的內部錯誤**(policy檔案壞掉、shell解析器crash、hook逾時)：這是完全不同的問題,講的是「保護機制本身失效時怎麼辦」。兩個Agent官方的hook機制都是fail-open(逾時或非2的結束碼一律放行)——這是外部給定的事實,Damping改變不了。因此Damping的責任是：**內部錯誤發生時,盡力寫一筆標記為`degraded`的稽核紀錄,並讓`damping doctor`能主動檢查「最近是否有degraded事件」**,把「無感」的代價控制在「保護暫時失效」而不是「保護暫時失效還完全不知道」。

### 對外比較誠實原則

寫進`docs/cli-reference.md`與README：任何提到dcg/Aegis/Pipelock等對手時,先承認對方做得好的地方,再指出結構性差異(跨channel、單一稽核出口)。不誠實的比較在開發者社群裡比不比較更傷品牌。
