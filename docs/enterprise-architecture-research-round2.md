## 2. Phase 3 — OAuth Provider

### Final verdict
**Round 1's core recommendation does not survive.** The load-bearing premise — "no vendor cleanly supports RFC 8707 + RFC 8693 together, so self-hosting is unavoidable regardless of buy-vs-build" — is factually false. **Authlete**, an established OAuth/OIDC-as-a-service vendor (10+ years, real enterprise references including NRI SecureTechnologies and Seven & I Holdings) documents RFC 8693 token exchange with an explicit `resource` parameter tying it directly to RFC 8707, out of the box, since Authlete 2.3. It wasn't in round 1's comparison table at all. It also offers self-hosted/on-prem license deployment, which partially satisfies Tim's self-hosting preference without requiring Damping to hand-roll audience-binding logic.

**New default path:** Trial Authlete as the primary candidate for the Gateway's token-issuance/audience-binding engine. Keep WorkOS AuthKit as the separate enterprise SSO front door (unchanged from round 1) — but note WorkOS *also* independently supports RFC 8707 for MCP already, so spike how WorkOS and Authlete (or whatever AS is chosen) compose before writing integration code; there may be overlap to resolve rather than two cleanly separate layers.

**Fallback, not default:** If Authlete fails on commercial terms, data residency, or contract fit, build a custom Fosite-based token-exchange handler. Budget this as real, uncounted engineering time — it is not "free because OSS," a point the research materially corrected (see below).

**Reject as primary:** Zitadel (RFC 8707 rejection confirmed still open, `github.com/zitadel/oidc#794`, no maintainer commitment) and Keycloak (JVM/ops burden is real, but see TCO correction below).

**New parallel workstream:** MCP's "Enterprise-Managed Authorization" extension went stable June 18, 2026 — built on the IETF ID-JAG draft chaining RFC 8693 + RFC 7523 — already adopted by Anthropic, Microsoft, and Okta, letting an enterprise's own IdP grant zero-touch MCP access. This is genuinely new information round 1 missed entirely despite being dated before its research window closed. It matters because it changes what Damping might actually need to *be*: for enterprise customers, the expected integration point may become "Damping interoperates with the customer's existing IdP via ID-JAG" rather than "Damping runs its own full OAuth AS that enterprise customers point at." This should be spiked before Phase 3/4 architecture is locked, not treated as a footnote.

### Why (incorporating verify + adversarial findings)

- **Ory Hydra's RFC 8693 support is not in the free product.** Ory's own current feature-comparison table marks Token Exchange "No" for OSS, "Yes" only for the paid Ory Enterprise License (launched April 2025) or Ory Network SaaS. Confirmed at the source level: the open Fosite library has no token-exchange handler file at all, zero code search hits. Round 1's table calling this "Documented, supported" was simply wrong for the product the recommendation was actually built on.
- **Hydra's RFC 8707 mechanism also isn't RFC 8707.** It's a proprietary `audience` query parameter requiring client pre-registration and custom consent-app code — a more static design than per-request resource targeting, and worse-fit for MCP where the downstream resource isn't known at client-registration time. Ory has zero tracked GitHub issues for RFC 8707, unlike Zitadel and Keycloak, which both do.
- **Hydra is not exempt from urgent-patch burden.** CVE-2026-33504 (CVSS 7.2, SQL injection in Admin APIs via forged pagination tokens) undercuts the "Go-native ⇒ lighter CVE burden than Keycloak" argument — a small team self-hosting Hydra still owns same-week emergency-patch obligations.
- **The Keycloak TCO figure was never independent.** The $142k-$211k number traces to Sirius Open Source, a vendor that sells managed Keycloak hosting and explicitly pitches itself as cheaper in the same post. A second vendor (FusionAuth, a direct Keycloak competitor) reaches a materially different $315,960 figure. Neither discloses reproducible methodology. Present this going forward as "a wide, vendor-sourced, commercially-motivated range ($142k–$316k)," not a credible independent estimate — the directional point (Keycloak's JVM/ops burden is heavier) still holds, the dollar figure does not.
- **The audience-binding gap this recommendation hinges on is currently semi-moot in practice.** Claude.ai itself doesn't send the RFC 8707 `resource` parameter yet, forcing an "audience validation disabled, fall back to client_id checks" workaround regardless of which AS sits underneath — meaning the specific technical differentiator (owning the audience-binding code) matters less near-term than round 1's framing implied.
- **Bandwidth argument, independent of any specific vendor:** multiple independent sources converge on "don't build/operate your own auth unless auth is your core product" — the strongest counter-evidence that self-hosted OSS auth servers operate well in production (Cloudflare's Hydra upgrade, OpenAI's 800M-user Hydra deployment) comes exclusively from organizations with platform/security engineering teams an order of magnitude larger than Damping. This is evidence about who has resources to do it well, not evidence a small team will.

**One correction in round 1's favor, worth stating plainly:** no evidence surfaced of a specific company that self-hosted plain Hydra (as opposed to full Ory stack or Keycloak) and publicly regretted it — this is a documentation/premise refutation and a bandwidth argument, not a smoking-gun horror story. The honest state is "not yet settled, tilted toward buying," not a confirmed self-hosting mandate.

## 3. Phase 4 — Cloudflare + SSO

### Final verdict
**The bottom line survives: stay on Cloudflare (Workers+Pages+D1+KV+R2), buy WorkOS AuthKit as the primary enterprise SSO vendor.** But round 1 overstated its case in two specific, checkable ways and underweighted a third — these are real amendments, not just caveats.

**Amendment 1 — Ory Polis exists and round 1's claim about it was wrong.** Round 1 stated self-hosting Ory for SAML "quietly concedes the buy argument" because real multi-IdP coverage requires Ory's paid enterprise tier (~$3k+/mo). This is false: **Ory Polis** (formerly BoxyHQ SAML Jackson) is Apache-2.0, genuinely free, lightweight (5-15MB binaries, no JVM), and already has named, mature support for exactly the IdP list cited (Okta, Entra ID, ADFS, OneLogin, PingOne, Google, JumpCloud), with real production integrations (NextAuth/Auth.js, Cerbos, Formbricks, Bubble). Add it to the Phase 4 comparison table as a named, reserved opt-out for cost-sensitive or maximally self-hosting-averse-to-vendor-lock-in customer segments — not the primary path, but no longer dismissible.

**Amendment 2 — Downgrade Clerk further, not less.** Round 1 treated Clerk as "cheaper fallback" with one notable Feb 2026 outage. The actual record is worse: three separately-acknowledged production outages in five weeks (Feb 10, Feb 19, Mar 10, 2026), with Clerk's own postmortems using language like "we have failed at our commitment to customers," plus a continuing cadence of 4-6 further incidents per month through July 2026. This strengthens, not weakens, the case for WorkOS as primary and demotes Clerk to "not recommended without re-checking its live incident history (`statuspage.incident.io/clerk/history`) immediately before any use."

**Amendment 3 — Pull the customer-owned audit-export pattern forward into Phase 4 now.** Cloudflare's cited outages hit the exact stack Phase 4 proposes building on more directly than round 1's brief date-list conveyed — Workers KV, D1, and Durable Objects were repeatedly and specifically named, including a Jul 3, 2026 incident that directly hit D1/Durable Objects only days before this research pass. For a security product whose audit trail sits on that stack, this is compliance-relevant, not cosmetic. Recommend designing a Teleport-External-Audit-Storage-style customer-owned export path (originally scoped for Phase 5) as part of Phase 4's initial design, not deferred.

**Secondary corrections (lower stakes, still worth fixing in the doc):**
- Cloudflare's "Code Orange: Fail Small" resilience program is no longer just announced — it completed on schedule (May 2026) with concrete shipped deliverables (progressive-rollout system, backup auth pathways for 18 services, mandatory code-review codex, a 200-person drill). It has not yet been tested by a real large-scale incident since completion. Keep the "decouple from dashboard availability" design regardless — the amendment is to the confidence level, not the mitigation.
- SushiData's "migrated off Cloudflare" is more precisely "migrated off D1 onto raw Durable Objects, stayed on Cloudflare" — a *more* Cloudflare-favorable data point than round 1's phrasing implied, and a proven within-platform escape hatch if the per-tenant D1 hot-window design is ever outgrown. Turso is a second, narrower watch-item for the D1 layer specifically — not a reason to move now.
- WorkOS itself had a critical SAML CVE (SAMLStorm, CVE-2025-29774/29775, CVSS 9.3) and a real incident history — buying WorkOS trades one vendor's availability/security risk for another, it doesn't eliminate vendor dependency. Apply the same "decouple from vendor availability" discipline to WorkOS that round 1 already applied to Cloudflare.
- WorkOS AuthKit already implements RFC 8707 for MCP in its live FastMCP integration — cross-reference this directly against the Phase 3 decision above.

## 4. Phase 5 — Enterprise Control-Plane

### Final verdict
**Round 1's specific build recommendation does not survive and should be reversed.** The proposal to ship one codebase in two deployment modes (self-hosted license-gated + Damping-hosted SaaS) *simultaneously, from the start of Phase 5* — including a newly-built key-rotation/Ed25519/rollback safety net — is not what any of round 1's own cited comparables actually did.

**New recommendation: SaaS-first, portable-by-design, self-hosted-mode deferred.**
1. Ship the Phase 5 control plane (policy authoring, fleet management, SSO, signed-bundle distribution, audit rollup) as **Damping-hosted SaaS only**, first.
2. Design it **portably** — no hard-wired Cloudflare-account-specific assumptions baked into core control-plane logic; keep OPA bundle handling as an injectable/swappable library. This is cheap now, expensive to retrofit later, and is the one part of round 1's instinct that genuinely holds up.
3. **Defer** the license-gated self-hosted mode, and its key-rotation/multi-key-signing/versioned-rollback subsystem, until a real enterprise or regulated prospect actually requires it — treat it as a scoped follow-on project triggered by demand, not a day-one build requirement.

### Why

- **Every comparable round 1 cited reached dual-mode years after founding, with much larger teams, generally only after real enterprise traction existed — not from day one.** Styra — the literal commercial company built around OPA, the closest analog to Phase 5 — launched cloud-only Styra DAS years after OPA's open-sourcing, and only added a distinct self-hostable control plane much later. GitLab's license-gated Enterprise Edition was added roughly two years after its Community Edition had already proven adoption. A near-exact modern analog, **Documenso** (open-source e-signature SaaS, founded 2023), shipped cloud SaaS + open self-hostable core together at launch but only introduced its license-gated Enterprise Self-Hosted tier in September 2025 — about two years later — explicitly because they'd started seeing a clear pattern of enterprises self-hosting on their own, i.e., built reactively once real demand existed, not preemptively. Damping has zero confirmed enterprise deals for this phase today.
- **"Reuse OPA's bundle format, near-free" understates the real lift.** OPA's own maintainers have had multi-key signing/rotation as an open feature proposal since October 2020 that was never natively shipped by the OPA project itself — a strong signal this is a genuine subsystem (signing service, rotation workflow, versioned registry, rollback automation, approval gating), not a bolt-on, even for teams with far more resources than Damping's.
- **Operators who actually run this pattern report it's a chronic tax, not a one-time cost.** Lago (open-source billing SaaS) describes self-hosted customer instances as "a black box" where routine hygiene changes can break a customer's production run, forcing permanent "bridge version" discipline. Supabase — a far better-funded team than Damping's — still has open, unresolved self-hosting pain in 2026 (inconsistent images, silently-failing migrations, "upgrade anxiety"). For a team Damping's actual size (effectively ~1 engineer plus AI assistance), this is an existential opportunity cost, not a marginal one.
- **Damping's own direct competitors haven't converged on simultaneous dual-mode either.** MintMCP is SaaS-first/managed with self-hosting as a separate later story; Pipelock (Lasso Security) is self-hosted-only with no SaaS control plane at all. The category hasn't converged the way round 1's comparable set (drawn from older, larger-category products) implied.
- **Styra should be dropped as the role-model exemplar.** Apple acqui-hired Styra's founders and core OPA maintainers (Tim Hinrichs, Torin Sandall, Teemu Koponen) in August 2025; Styra's commercial DAS product is being sunset, with customers told to migrate to alternatives (e.g., Axiomatics). `docs.styra.com` and `www.styra.com` are confirmed dark — traced to a specific, verifiable technical cause: DNS delegation for `styra.com` now includes Apple's own nameservers, which refuse to answer for the zone (a "lame delegation," independently confirmed via direct DNS resolution and IP-ownership lookup). This is a company whose hybrid open-core business did not survive independently — a cautionary tale, not proof the pattern works commercially. The underlying engineering premise (reuse OPA's signed-bundle wire format) is *unaffected, arguably strengthened* since it's now a first-party, Apache-2.0, CNCF-governed feature of the OPA project itself ("OPA Control Plane," actively released as recently as June 2026) rather than tied to one vendor's fortunes — cite that project, not Styra DAS, going forward.
- **The competitive landscape is less empty than round 1 assumed.** CodeIntegrity ($4.8-5M seed, May 2026) ships a self-hostable "Zero Trust Control Plane" for agent/MCP tool-call governance — close enough to Damping's positioning to warrant its own dedicated competitive look (pricing, actual customers, exact deployment split) before Phase 5 messaging is finalized. ToolHive/Stacklok shows a real open-core precedent in adjacent MCP-security tooling, though it chose self-hosted-only (no SaaS), a different split than Damping is considering. Anthropic's own native Claude Code enterprise admin controls (Managed Settings, Compliance API, expanded April 2026) are a platform-vendor risk to the cross-agent value proposition worth factoring into differentiation, separate from third-party competitors.
- **Teleport's External Audit Storage pattern is confirmed real and works as described** (customer-owned S3, Teleport-as-OIDC-provider, AWS-only, one documented Access-Monitoring interaction gap) — a good template to borrow for the audit-export design now pulled forward into Phase 4 (see above), and later for Phase 5's eventual self-hosted mode.

## 5. Phase 6 — Memory & Context Poisoning Defense (including Skills)

### Final verdict
**Round 1's core design is confirmed sound and extends cleanly to Skills' write-time half — but Skills are a genuinely distinct, larger attack surface that needs new schema elements round 1 didn't anticipate, not just a wider glob list.**

Confirmed unchanged: widen `cli/cmd/hook.go`'s filter beyond `Bash` to classify `Edit`/`Write`/`MultiEdit` writes to known memory-store paths as a new `event.ActionMemoryWrite` type; layer provenance tagging + deterministic trust scoring + structural pattern detection on top; surface findings as a non-blocking `ReviewRequired` flag, never a new blocking verdict, to avoid the "nagging → uninstalled" failure mode. The retrieval/context-assembly-time correlation attack (poison now, triggered weeks later via unrelated context) remains correctly flagged as not solvable unilaterally.

**What Skills add, and why it's not "the same problem via a different file path":**

1. **Write-time extension (additive, low-risk):** broaden the memory-store path-glob config to a multi-tool union — `.claude/skills/**`, `~/.claude/skills/**`, `.cursor/skills/**`, `.agents/skills/**` (Codex), plugin `skills/` directories, nested per-subdirectory variants — and, critically, match **any file under a matched skill directory**, not just files literally named `SKILL.md`. Anthropic's own engineering blog confirms skills can bundle scripts Claude runs "without loading either the script or the [data] into context" — a bundled `scripts/*.sh` payload never appears as text any NL-pattern matcher would ever see, so the content-hash/diff mechanism must cover non-markdown files too.
2. **New invocation-time action type — genuinely new, no memory-file analogue:** add `event.ActionSkillInvoke`, hooking `tool_name: "Skill"` PreToolUse events (model-invoked skills only — a human typing `/skillname` bypasses PreToolUse, correctly, since hooks gate agent autonomy not explicit human commands). Its value is *correlation*: if a specific skill was flagged at its most recent write, its invocation becomes a second, later, independent signal attached to `Decision.RelatedEventIDs` — converting "was this instruction read into context" (invisible, for CLAUDE.md) into "was this exact named skill executed" (visible, for Skills) — a strictly stronger signal than plain memory poisoning offers. This needs empirical confirmation of the actual `tool_input` payload shape via a live-session BDD test before implementation, consistent with this project's own verification discipline.
3. **Two new categorical non-goals, stated as plainly as the existing retrieval-time gap — not folded into it:**
   - **Dynamic context-injection preprocessing** (the `` !`command` `` syntax in a SKILL.md) executes *before* Claude or any hook pipeline sees anything. Datadog Security Labs built a working exploit PoC (`` !`gh auth token > token` `` + curl exfiltration) that completed before any model-level refusal fired. Anthropic's only documented control is a blunt global kill-switch (`disableSkillShellExecution`), which itself implies this path sits outside anything hook-based architecture can observe. Damping's only lever is documenting the recommendation and checking for it in `damping doctor` — not interception.
   - **Cloud-account-synced skills** (claude.ai Skills, "Cowork" platform plugins) are confirmed, via a maintainer-closed GitHub issue, to inject into every session with **zero local file write and zero user-facing control** — categorically prior to the retrieval-time gap, since Damping can't observe a write that never happens on disk, no matter how the `hook.go` filter is widened. This may not be solvable by the user either, not just by Damping.
4. **Cross-tool blast radius:** Cursor (v2.4, Jan 2026) and Codex both adopted the same open `agentskills.io` SKILL.md standard and cross-load `.claude/skills/`, `.agents/skills/` — a single poisoned skill in a shared repo has cross-tool reach that CLAUDE.md/.cursor/rules never had. Cursor's `afterFileEdit` hook is confirmed real but post-hoc/observation-only (no pre-write block) — fine for the Review-only design, it just determines which hook event feeds the Cursor adapter.
5. **Self-modification risk, but existing mitigation applies:** skill-scoped `hooks:` frontmatter can rewrite Claude Code's own permission settings (PromptArmor documented this concretely) — this is exactly the "agent discovers and removes its own safety layer" scenario threat-model.md already names, with an existing mitigation (`damping doctor`'s hook-registration diff). Worth being explicit that this is detect-after-the-fact, not prevention, and that a poisoned Skill is now a concrete, automatable trigger for that scenario.

**New open questions to append to round 1's list:** exact `tool_input` payload shape for Skill invocation (needs live BDD confirmation); whether bare (non-plugin) skill `hooks:` get the same trust-dialog friction as plugin-wrapped ones; Cursor/Codex hook parity for skills is asymmetric and only partially investigated; false-positive risk is plausibly *higher* for Skills than plain memory writes, since Skills are an actively-promoted, legitimate power-user feature Damping's own target users are likely building at the exact moment a false positive would fire.

## 6. What Round 1 Got Wrong or Oversold

Being direct, since Tim explicitly asked for this rather than defense of the prior work:

- **Phase 3 — the most significant miss.** Round 1's central premise ("no vendor supports both RFCs") was simply false; Authlete was missing from the comparison table entirely. Round 1 also mischaracterized Ory Hydra's RFC 8693 support as "documented, supported" when it is gated behind a paid Enterprise License in the actual open-source product the recommendation was built on — this is not a minor hedge miss, it's the load-bearing technical claim being wrong. The Keycloak TCO figure was presented with more authority ("independent estimates") than it deserved — it traces to a Keycloak-hosting competitor's blog with an undisclosed methodology, and a second vendor's estimate disagrees by over $100k. Round 1 also missed a newly-relevant 2026 industry development (MCP's EMA/ID-JAG extension) that predates its stated July 2026 compilation date.
- **Phase 4 — one clean factual error, one underweighting.** The claim that self-hosted Ory "can't do SAML without paying the enterprise tier" is false (Ory Polis/BoxyHQ Jackson exists, is free, and covers exactly the IdP list cited) — and it's an internally inconsistent standard, since round 1 used this exact "quietly concedes the argument" logic to reject Ory for Phase 4 but never applied the same test to Hydra in Phase 3. Round 1 also underweighted how directly Cloudflare's cited outages hit the specific services (D1, Workers KV, Durable Objects) Phase 4 proposes building on, rather than more tangential CDN products, and slightly softened how bad Clerk's actual reliability record is.
- **Phase 5 — the second most significant miss.** Recommending simultaneous dual-mode build from day one contradicts the very comparables round 1 cited to justify it — none of them (Styra, GitLab, or the closer analog Documenso) actually built that way; they all sequenced SaaS/simple-self-host first and added license-gated enterprise self-hosting years later, after real demand. Round 1's own text even contained the qualifier ("converges on the hybrid pattern once it serves enterprise") but didn't apply that timing lesson to Damping's own current stage. Holding up Styra as a role model was also simply stale — the underlying business had already effectively ended (Apple acqui-hire, August 2025) before or during round 1's research window.
- **Phase 6 — not wrong, but scoped too narrowly.** The design implicitly treated agent "memory" as inert text content, which is true of CLAUDE.md/.cursor/rules but not of Skills, which carry real execution primitives (opaque bundled scripts, pre-model preprocessing, self-modifying permission hooks) memory files structurally lack. This wasn't a factual error so much as an incomplete threat-surface inventory at the time.

## 7. Remaining Genuine Uncertainties

These do not resolve with more research — they need a real spike, a vendor conversation, or a decision only Tim can make about capacity and business direction.

- **Phase 3:** Whether Authlete's actual commercial terms (pricing, contract structure, data residency) fit Damping's business model requires a real vendor conversation and trial, not more searching. Whether the MCP EMA/ID-JAG pattern becomes a broad industry norm or stays Okta/Anthropic/Microsoft-specific over the next 6-12 months can only be watched, not predicted. Whether building a custom Fosite token-exchange handler is realistically within near-term team bandwidth given everything else on the roadmap is a prioritization conversation with Tim, not a technical question.
- **Phase 4:** Whether Damping's actual buyers care enough about vendor-neutrality to value Ory Polis as a real option (versus it being an unused checkbox) is a go-to-market question that depends on who buys the product, not a research question. Whether Cloudflare's "Code Orange" resilience work holds up under the next real multi-service incident cannot be known until it happens — the only actionable response is the design mitigation (decouple, export audit data), already recommended.
- **Phase 5:** The actual trigger condition for building the deferred self-hosted mode is landing a real enterprise or regulated prospect that requires it — that's a sales and market-timing question, not an engineering one, and no amount of research substitutes for having that conversation happen. Whether CodeIntegrity, a better-funded competitor, or Anthropic's own platform-native governance features close the differentiation gap faster than Damping can build is an ongoing competitive uncertainty that needs continuous watching, not a one-time answer.
- **Phase 6:** Whether the two newly-identified "unsolvable" Skills gaps (dynamic context-injection preprocessing, cloud-account sync) are permanent architectural facts or just current product gaps Anthropic might expose a hook for later is genuinely unknowable without engaging Anthropic directly or watching their changelog — this may be worth raising with Anthropic alongside round 1's already-flagged hook-protocol feature request. The exact payload shape for Skill invocation events needs a live BDD spike against a real Claude Code session, not another documentation read.
- **Cross-cutting, and probably the single biggest constraint:** Damping's actual team size and bandwidth threads through every phase above (build-vs-buy in Phase 3, self-host-vs-buy SSO in Phase 4, dual-mode timing in Phase 5). No research resolves how much a team Damping's size can realistically take on simultaneously — that is a personal capacity and business-prioritization decision only Tim can make, and it should probably be made explicitly, once, and then used to gate all four phases consistently, rather than re-litigated per phase.
